# 0011 - Site generation architecture: Python generator emitting committed Markdown into Starlight

## Status

Accepted

## Context

The documentation site is migrating from MkDocs Material to Astro Starlight (see the design spec at `docs/superpowers/specs/2026-05-31-astro-starlight-site-design.md`). The goal is a rich, browsable library that surfaces the catalog's 60 entries, 195 examples, 12 diff-pairs, 5 recipes, and 15 format templates - especially the examples, which are currently orphaned (only 1 of 60 entries links to its examples).

The architecture question was how the site turns the source catalog (`taxonomy/`, `examples/`) into pages. The maintainer framed it as a family standard, not a one-repo choice: "reuse the existing Python tooling unless another approach is genuinely more effective and efficient, in which case the sibling repo pm-skills should adopt it too." Three candidates were evaluated:

- **A - Python generator emits committed Markdown/MDX into the Starlight `docs/` tree.** Reuses the existing Python frontmatter tooling (`validate.py`, `build-indexes.py`); matches pm-skills.
- **B - Astro Content Layer loaders** read `taxonomy/`/`examples/` directly as typed collections, rendered by components, with no intermediate committed markdown.
- **C - Node/TS generator** emitting markdown like A but in the site's own language.

A dedicated multi-agent investigation (three parallel investigations plus adversarial synthesis, 2026-05-31) verified the deciding facts against current Astro 6.3 and Starlight 0.39 documentation via Context7.

## Decision

Use **approach A**: a Python generator emits committed Markdown/MDX into the Starlight `docs/` tree, where Starlight's native autogenerate sidebar renders it. Narrative pages stay hand-authored. The generator emits Starlight-correct slug links in-generator (no post-build link rewriter), uses PyYAML (not the repo's hand-rolled parser), and is guarded by a regenerate-and-`git diff` freshness check in CI.

Reject **B** and **C**.

## Consequences

### Positive

- **Native sidebar navigation.** Generated markdown lands in `docs/`, where Starlight's `items: [{ autogenerate: { directory } }]` sidebar works with no custom infrastructure.
- **Reuses tested Python tooling** and keeps the content pipeline in one language; matches the pm-skills family precedent, so the two repos stay converged with no migration to fund.
- **Committed, reviewable pages**: greppable, PR-reviewable, GitHub-browsable, with a freshness guard proving they match source.
- **Richness is fully achievable**: tabbed-by-topic examples (built-in `<Tabs>`), side-by-side diff-pairs (one `DiffPair.astro` fed by a header split), galleries (`Card`/`CardGrid`), and automatic copy buttons all render from generated MDX importing components.

### Negative

- **Committed generated files** produce larger diffs on catalog edits, and require a freshness guard and the discipline of regenerating after source edits.
- **Two-language pipeline** (Python generator + Node site shell) is more surface than a single-language setup.
- **Verbatim source prose** means source defects render live unless caught by a content-fidelity lint.

### Neutral / rejection rationale

- **B is rejected on a verified blocker, not a preference.** Starlight 0.39 cannot place Content Layer pages into an autogenerated sidebar group: per current Starlight docs (`guides/pages.mdx`), `<StarlightPage />` pages "are not part of a collection and cannot be added to an autogenerated sidebar group," the `sidebar` frontmatter is unavailable, and the per-page `sidebar` prop overrides the whole sidebar rather than joining the autogenerate tree. `docsLoader()` is a fixed glob wrapper (base pinned to the docs dir, only `generateId` overridable), so the docs collection cannot read `taxonomy/`. B would force ~90 hand-maintained sidebar links or a custom, unsupported Starlight integration. Richness, B's only candidate advantage, is a wash (A reaches every feature via MDX components), so nothing offsets the navigation regression.
- **C is rejected as a strictly-worse A**: it inherits all of A's committed-file, freshness, and link costs, gains no rendering or sidebar advantage, and discards the existing Python tooling and the family precedent.
- **pm-skills should not migrate to B.** The family is already converged on A; the "switch only if clearly better" bar is not met. The one cross-repo convergence worth pursuing is independent of A-vs-B: emit Starlight-correct slug links in-generator so pm-skills can retire its 238-line post-build link rewriter.

### Durability note

This ADR exists specifically to pre-answer a likely future temptation to "modernize" to Content Layer for elegance. The blocker (no supported sidebar integration in Starlight 0.39) is the load-bearing reason; revisit only if a future Starlight release adds a supported way to inject autogenerated sidebar nodes from a custom collection.
